App Service 的并发模型

一个 App Service instance = 一个跑着你 web server 的进程（你的 case 是 uvicorn）。它同时 serve 所有 10 个用户，每个 HTTP 请求完全独立。

  • 如果流量大到一个 instance 扛不住，App Service 会横向扩出更多 instance（scale-out），每个 instance 照样并发 serve 多人。

是的，OBO 就是用前端传下来的 identity。具体讲：

EasyAuth 登录之后，会往 header 里塞一个 X-MS-TOKEN-AAD-ACCESS-TOKEN —— 这个 token 是 Azure AD 发给"你这个 App Service"的，scope 是你的 app。它不能直接用来调 ADF，因为 scope 不对。

Device login 和 OBO 是二选一的两条路：

大概长这样：

# agent_runtime/core/auth.py
@dataclass
class TokenRecord:
    token: str
    expires_at: float   # unix timestamp
    provider: str       # "azure", "snowflake", "ado"...

_cache: dict[tuple[str, str], TokenRecord] = {}   # (user_id, provider) -> record
_lock = threading.Lock()

def get_token(user_id: str, provider: str) -> str | None:
    with _lock:
        rec = _cache.get((user_id, provider))
    if rec and rec.expires_at > time.time() + 60:   # 留 60s buffer
        return rec.token
    return None   # caller 需要触发 refresh / device login

def put_token(user_id: str, provider: str, token: str, expires_at: float):
    with _lock:
        _cache[(user_id, provider)] = TokenRecord(token, expires_at, provider)

调用路径：

tool call 进来 → runtime 从 ContextVar 读 user_id
              → get_token(user_id, "azure")
              → 有 token → 注入到 MCP tool args → 执行
              → 没 token → 触发 device login / OBO → 拿到后 put_token → 再执行

要注意的几点

  • 进程重启 = token 丢失：用户得重新 login / 重做 OBO。对于短 session 的 chat 场景没问题。
  • 多 worker：uvicorn 跑多 worker 时每个 worker 有自己的 dict。如果同一个用户的请求被路由到不同 worker，第二个 worker 得重新取 token。可以接受（OBO 很快、device login 是 once-per-session）。
  • 多 instance App Service（scale-out）：同样问题放大。如果你 scale 到多实例，要么开 sticky session（简单），要么把 cache 换成 Redis/Key Vault（正经）。MVP 阶段 sticky session 够了。
  • 锁：dict 读写加 threading.Lock。因为 uvicorn 用 thread pool 跑 agent loop（你 engine 里的 ThreadPoolExecutor），多线程并发读写同一个 dict 理论上 CPython GIL 保护，但别赌，加锁是干净的。

这个问题的本质是："container 既被 EasyAuth 挡着的 UI 调，也被其他 service 直接调"。两种流量形态不能走同一套"信 header"的逻辑，但可以走同一套验签的逻辑。

正解：别信 header，验 JWT

X-MS-CLIENT-PRINCIPAL-* 是 EasyAuth 给你的便利信息，不是凭证。真正的凭证是 JWT access token / id token —— 它带 Azure AD 的签名，任何人都伪造不了（前提是 Azure AD 私钥没泄漏）。

EasyAuth 实际上同时给你塞了两样：

  • X-MS-CLIENT-PRINCIPAL-*（解好的 claims，方便但不能信）
  • X-MS-TOKEN-AAD-ID-TOKEN / X-MS-TOKEN-AAD-ACCESS-TOKEN（原始 JWT，带签名）

只用后者。这样不管流量从哪来，auth 逻辑都一样。

统一的 auth dependency（container 一份代码，两种流量通吃）

# agent_runtime/api/auth.py
from jwt import PyJWKClient, decode as jwt_decode

TENANT_ID = os.environ["AAD_TENANT_ID"]
AUDIENCE  = os.environ["AAD_AUDIENCE"]   # 你这个 app 的 client_id 或 API URI
_jwks = PyJWKClient(
    f"https://login.microsoftonline.com/{TENANT_ID}/discovery/v2.0/keys",
    cache_keys=True, lifespan=3600,
)

async def require_user(request: Request) -> UserIdentity:
    # 统一入口：Authorization: Bearer <jwt>
    # - UI 流量：frontend 从 EasyAuth 的 X-MS-TOKEN-AAD-ID-TOKEN 拿出来，再以 Bearer 塞给 runtime
    # - Service 流量：caller 自己带 Bearer token
    auth = request.headers.get("Authorization", "")
    if not auth.startswith("Bearer "):
        raise HTTPException(401, "missing bearer token")
    token = auth[7:]

    try:
        key = _jwks.get_signing_key_from_jwt(token).key
        claims = jwt_decode(
            token, key,
            algorithms=["RS256"],
            audience=AUDIENCE,
            issuer=f"https://login.microsoftonline.com/{TENANT_ID}/v2.0",
        )
    except Exception as e:
        raise HTTPException(401, f"invalid token: {e}")

    return UserIdentity(
        user_id=claims["oid"],
        principal_name=claims.get("preferred_username") or claims.get("upn", ""),
        display_name=claims.get("name", ""),
        aad_access_token=token,
        is_authenticated=True,
    )

验 4 样：签名、iss（发行方是你的 tenant）、aud（受众是你这个 app）、exp（没过期）。缺一不可。

两种流量怎么走

UI 流量（浏览器 → Front Door/App Service → EasyAuth → frontend → runtime）：

agent_frontend/server.py 做两件事：
  1. 从 X-MS-TOKEN-AAD-ID-TOKEN 里读出 JWT（EasyAuth 给的）
  2. 转发到 runtime 时，写成 Authorization: Bearer <jwt>

agent_runtime 侧：require_user 验签 → 通过

Service 流量（别的服务直接调 runtime）：

Caller 自己用 client_credentials / managed identity 去 Azure AD 拿 token（aud = runtime）
  → Authorization: Bearer <jwt> 打给 runtime
  → require_user 验签 → 通过

攻击者伪造 X-MS-CLIENT-PRINCIPAL-* 打进来：

require_user 根本不看这些 header，只看 Bearer token
  → 没 token / 假 token → 401

三层防御（按投入排）

1. 必做 — 应用层 JWT 验签（上面那段代码）。有了这个，就算别人绕过 EasyAuth 直接打到 container，也进不来。这是逻辑上的保护。
2. 强烈建议 — 网络层限制（App Service Access Restrictions / Private Endpoint / VNet integration）。让 container 物理上只接受来自 Azure Front Door / 特定 VNet 的流量。defense in depth。
3. 可选 — "Require Auth" 模式。App Service 的 EasyAuth 有一个开关叫 "Require authentication"：未登录直接 302 到 AAD，不会走到你的 app。这是针对 UI 路径的硬门禁。对 service 路径你可以选择关掉 EasyAuth 或让它 pass-through。

Option A（MVP 推荐）— 一个 AAD app

Option B 你付出的代价

1. 多注册一个 AAD app（runtime-app），并在"Expose an API" 里定义一个 scope（比如 access_as_user）
2. 在 AAD 里配 permission：frontend-app 需要 "API permissions" → 勾选 runtime-app 的 scope，管理员 consent 一次
3. frontend 需要 client_secret 才能跑 OBO（反正做 EasyAuth 已经有了，不算新增）
4. 每个用户 session 首次 打 runtime 时多一次 OBO 调用（~50ms，之后 cache 掉）
5. 配置文档变长，新同事部署要多做几步

Option B 你换到的好处

1. token 不能跨层重放

  • A：偷到 frontend 的 cookie/token → 直接能打 runtime
  • B：偷到 frontend 的 token → aud=abc，runtime 验签 aud==xyz 直接拒绝

2. 访问控制分层

  • A：谁能用 frontend = 谁能用 runtime，一视同仁
  • B：可以在 AAD 里给 runtime-app 单独加一组 "允许访问" 的 user/app（比如某个调度服务只给 runtime 权限，不给 frontend 权限）

3. Service caller 语义干净

  • A：另一个后台服务想直接调 runtime → 得装作自己是 frontend（去要 aud=abc 的 token），违和
  • B：那个服务去 AAD 要 api://runtime/.default 的 token → aud=xyz → 打 runtime，天经地义

4. audit log 能分清楚来源

  • 看 runtime 的访问日志，token 的 azp（authorized party）字段会告诉你是 frontend-app 还是其他 app 调进来的

5. 符合"最小信任"原则

  • frontend 和 runtime 在 AAD 看来是两个独立的 resource，不是"同一个 app 的两个 endpoint"

什么时候从 A 升到 B

触发条件，任一满足就该升：

  • [ ] runtime 要开始接受 frontend 之外的 caller（就是你最初问题里"未来也能是一个 service"那个场景）
  • [ ] 安全 review 要求 audience 隔离
  • [ ] 你想给不同 user 群体开不同的 runtime 访问权限（不跟 frontend 绑定）
  • [ ] runtime 和 frontend 要部署在不同的 Azure 订阅/tenant
  • [ ] 多个 frontend（比如 web + 手机 app）共用一个 runtime

OBO 链条在两种方案下都一样

  • Service identity（container 自己是 actor）：Managed Identity / Service Principal / 固定 PAT
  • User identity（人类是 actor）：Device Code 登录拿 user-scoped token

4. Snowflake/ADO 本来就没有 Managed Identity 对等物，它们 native 就该走 user device code 或 container-owned service account。你的模型天然统一。
5. OBO 可以完全跳过，因为 Device Code 能覆盖它的所有场景：

  • 用户想以自己身份调 Azure → 做一次 Azure device code login
  • 用户想切特权号 → 做一次特权号的 device code login
  • UX 代价：首次登录多开一个 tab。省了：一整套 OBO 配置（API permissions、frontend client_secret 管理、OBO 失败的各种边界）

所以我推荐：v1 根本不做 OBO，只做 service credential + user device code 两种。要做 OBO 时再加，加回来也只是多一个 mode: "obo" 选项，不改架构。

这问得好，我没讲透。你一半对：identity 不再当 credential（token）用 —— 这部分你说对了。但它还在当 index（key）用，而且这个 index 如果错了会有严重的安全问题。

最要命的一个例子：device code cache 的 key

看 device_code 这条分支：

cache[(user, provider)]  ← 这个 key 里 "user" 从哪来？

假设你把 user 拿掉，cache 只按 provider 做 key：

时刻 T1:
  用户 A 进来 → 点了用 Snowflake 的 agent → cache miss
  → 触发 device flow → A 去新 tab 登录自己的 Snowflake 账号
  → 拿到 token → cache["snowflake"] = A 的 token ✅

时刻 T2（同一个 container，同一瞬间）:
  用户 B 进来 → 点了用 Snowflake 的 agent → cache hit
  → 用 cache["snowflake"] 调 Snowflake → ❌ B 正在用 A 的身份查 Snowflake

时刻 T3:
  B 在不知情的情况下看到了 A 的 Snowflake 数据
  A 在 Snowflake 的审计日志里背锅，B 做的查询记在 A 头上

这是 data leak。 必须 cache[(user_id, provider)] 才安全 —— 每人一把 token。所以 runtime 必须知道当前请求是谁。

Service mode 真的不需要吗？再看一下

Service mode 下，token 本身是 container 的，不挂在 user 身上。但这些问题还在：

1. SSE event routing：用户 A 正在 chat，某个 tool 需要 device flow（比如 agent 有多个 provider，一个走 service，一个走 device）→ device URL 要推给谁的 SSE 连接？必须按 user 路由。
2. Admission control：没 auth 的情况下，任何人能打到你的 /api/chat。service mode 只是解决了"用什么身份调下游"，没解决"谁能用这个 agent"。公司内部 agent 你总不能让非本公司人随便调吧。
3. Conversation 归属：agent_frontend 已经有 session 存储了，_DEFAULT_USER_ID="local" 那里总得填点啥 —— 不按 user 分，大家的 session 混在一起。
4. 审计："某用户触发了 ADF 查询"是合规要求，即使实际 token 是 service 身份。LangSmith metadata、access log 都要这个字段。
5. Rate limit / quota：每人每分钟 N 次请求 —— 没 user_id 就做不了。

那为什么要验 JWT，不是只读 header？

因为"知道 user_id"和"可信地知道 user_id"是两件事。

  • X-MS-CLIENT-PRINCIPAL-ID 这个 header 任何人都能塞（只要能打到 container）。你读出来做 cache key → 攻击者塞个别人的 user_id → 缓存污染，又变成 data leak。
  • 验 JWT 签名（用 Azure AD 公钥）是密码学证明"这个 user_id 确实是 Azure AD 发给这个人的"。伪造成本 = 拿到 Azure AD 私钥 = 不现实。

所以验 JWT 的目的不是"为了拿 token 下沉"，是"为了让 user_id 可信"。下沉 token 那一步我们已经砍掉了，但 user_id 作为 cache key 和路由 key 还必须可信。

换一个说法总结

之前我讲错了点什么：我把 identity 说得好像只是为了 OBO。砍掉 OBO 之后听起来确实像"那还验 JWT 干嘛"。但其实：

用途需要 identity 吗为什么
Downstream credential❌（已砍）OBO 砍了
Device code cache key✅不按 user 分桶 = 跨用户 token 污染 = data leak
SSE event routing✅device URL 得推给对的那个人
Admission control✅无 auth = 任何人可调
Audit / quota✅合规 + 防滥用

JWT 验签 = 让上面这些 "user_id" 可信。identity 不做 token 了，但依然是系统的分桶 key。

有没有办法连 JWT 都不验？

理论上可以，条件是：

  • 网络层已经锁死（App Service Access Restriction + Private Endpoint）—— 保证没有 EasyAuth middleware 的请求物理上打不到 container
  • 只相信 EasyAuth 注入的 header
  • 接受"以后想把 runtime 当 service 给别人调"时要重写这块

MVP 想快可以这么做，等于把 threat model 推给 infra 层。但代码里读 header 后那一刻就要提取 user_id 存起来，下游全用那个值。未来加 JWT 验签只改入口这一处。

所以这张架构图里 验 JWT → UserIdentity → ContextVar 这一步保留是对的，但语义从"拿 token 给下游用"变成"拿可信的 user_id 给 cache 和 routing 用"。要我把图的这一步标注改清楚吗？

两个决策各自的意义

A 决策（共用 vs 独立 AAD） 影响的是谁能打进 runtime：

  • A1 共用：runtime 的 audience 就是 frontend 的 audience —— 有 frontend token 的都能打 runtime。frontend 和 runtime 视为同一个 app。
  • A2 独立：runtime 是一个独立的 "resource"，在 AAD 里可以单独授权。别的 service 想直接调 runtime，去 AAD 要 aud=runtime 的 token 就行，不用装成 frontend。为"runtime 也要作为独立 service 对外"准备。

B 决策（OBO vs 不 OBO） 影响的是下游服务看到谁在调用：

  • B1 OBO：ADF 日志里是"用户 Alice 查询"。Alice 在 ADF 没权限就查不到。合规场景常要求这个。
  • B2 不 OBO：
  • Service (MI)：ADF 日志里是"container-X 查询"。Alice 能查的取决于 container-X 的 RBAC，所有用户看同样的数据。
  • Device code：ADF 日志里还是"用户 Alice 查询"，但 Alice 要额外登一次。和 OBO 效果一样，只是 UX 换工作量（无 OBO 配置换用户多一次点击）。

所以 A 和 B 正交：A 关心"进 runtime 要不要换票"，B 关心"调 ADF 要不要以用户身份"。

我们选 ① 的三个理由

1. B 决策（不 OBO）：Device code 已经能覆盖"以用户身份调下游"的刚需。Managed Identity 覆盖"共享数据视图"的常见场景。OBO 的唯一好处是无感，不值得这个复杂度。
2. A 决策（共用 AAD）：MVP 阶段 runtime 只被 frontend 调，没有"独立对外 service"的需求。多一个 AAD app = 多一份 client_secret 管理 + frontend 要多做一次 OBO。
3. 升级成本低：①→② 只需加一段 OBO 代码；①→③ 只需注册第二个 AAD app 并改 audience 配置；runtime 业务代码都不用动。所以一开始选最简的，有需要再升。

一句话判别该选哪个

  • 只有 frontend 调 runtime，数据访问共享就行 → ①（共用 + 不 OBO）
  • 只有 frontend 调 runtime，但合规要求 user-level audit → ②（共用 + OBO），或 ① + device code
  • runtime 未来要作为 API 给别的 service 调 → 升到 ③ 或 ④
  • 既对外开放 runtime 又要用户身份穿透到下游 → ④

你现在的需求是第一条，所以 ① 是对的。未来如果第三条成立（runtime 对外），直接跳 ③，仍然不用 OBO。

  • 10 个容器都跑同一份代码、同一个团队拥有、同种数据敏感级别 → 在 AAD 看来这是一个 logical service，一个 AAD app 就够
  • 10 个容器分属不同团队、不同数据合规等级、对外 API 契约不同 → 这是 10 个 service，才需要 10 个 AAD app
